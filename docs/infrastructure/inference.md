# 推理优化 {#inference}

推理是AI模型实际服务用户的过程，优化推理性能和成本是AI落地的关键。

---

## 推理 vs 训练

| 维度 | 训练 | 推理 |
|------|------|------|
| 目标 | 学习参数 | 使用参数 |
| 计算 | 前向+反向传播 | 仅前向传播 |
| 数据 | 大批量 | 单个/小批量 |
| 优化重点 | 吞吐量 | 延迟 |
| 成本 | 一次性 | 持续性 |

---

## 推理的挑战

### 1. 内存瓶颈
- 模型参数需要加载到显存
- 70B模型需要140GB（FP16）
- KV Cache随序列长度增长

### 2. 延迟要求
- 用户期望快速响应
- 首个token延迟（TTFT）
- 每个token延迟（TPS）

### 3. 成本
- GPU成本高昂
- 需要持续运行
- 成本与请求数成正比

## 量化（Quantization）

### 什么是量化？
将模型参数从高精度转换为低精度：

```
FP32 (32位) → FP16 (16位) → INT8 (8位) → INT4 (4位)
```

### 量化方法

| 方法 | 位数 | 精度损失 | 内存节省 |
|------|------|----------|----------|
| FP16 | 16位 | 无 | 2x |
| INT8 | 8位 | 很小 | 4x |
| INT4 | 4位 | 小 | 8x |

### 主流量化技术

#### GPTQ
- 训练后量化
- 逐层量化
- 支持4bit

#### AWQ（Activation-aware Weight Quantization）
- 基于激活值的权重量化
- 精度损失更小
- 4bit量化

#### GGUF（llama.cpp格式）
- 本地推理标准格式
- 支持多种量化级别
- CPU/GPU混合推理

#### bitsandbytes
- 8bit/4bit量化
- 与Hugging Face集成
- 支持QLoRA

## 推理引擎

### vLLM
- 最流行的推理引擎
- **PagedAttention**：高效KV Cache管理
- 支持连续批处理
- 高吞吐量

### TensorRT-LLM（NVIDIA）
- NVIDIA官方推理引擎
- 高度优化
- 支持各种量化

### llama.cpp
- C++实现的推理引擎
- 支持CPU推理
- GGUF格式
- 适合本地部署

### Ollama
- 简化本地LLM运行
- 基于llama.cpp
- 一行命令运行模型
- 类似Docker的体验

### SGLang
- 高效推理引擎
- 结构化生成优化

### TGI（Text Generation Inference）
- Hugging Face的推理引擎
- 生产级部署

## 推理优化技术

### 1. KV Cache
- 缓存注意力计算的Key和Value
- 避免重复计算
- 但占用大量内存

### 2. PagedAttention
- vLLM的核心技术
- 像操作系统管理内存一样管理KV Cache
- 减少内存浪费

### 3. 连续批处理
- 动态组批
- 提高GPU利用率
- 增加吞吐量

### 4. 推测解码（Speculative Decoding）
- 用小模型预测大模型的输出
- 大模型验证
- 减少延迟

### 5. Flash Attention
- 高效注意力实现
- 减少内存访问
- 加速计算

- 数据隐私

## 蒸馏（Distillation）

### 什么是蒸馏？
用大模型（教师）训练小模型（学生）：

```
教师模型（70B）→ 训练 → 学生模型（7B）
```

### 蒸馏的好处
- 小模型获得大模型的能力
- 推理成本大幅降低
- 适合边缘部署

### 代表
- DeepSeek R1蒸馏版本（1.5B-70B）
- Phi系列（Microsoft）

## 2025年推理趋势

1. **推理引擎标准化**：vLLM等成为标准
2. **量化普及**：4bit量化成为常态
3. **推测解码**：减少延迟的关键技术
4. **边缘推理**：更多模型在端侧运行
5. **推理成本下降**：竞争推动成本降低
