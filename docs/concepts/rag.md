# RAG（检索增强生成） {#rag}

RAG是解决AI"幻觉"问题的最重要技术之一，让AI能够基于真实数据回答问题。

---

## 什么是RAG？

**全称**：Retrieval-Augmented Generation（检索增强生成）

**一句话定义**：RAG让AI在回答问题前，先从知识库中检索相关信息，然后基于这些信息生成回答。

**核心思想**：不要让AI"凭记忆"回答，而是让它"查资料"后回答。

---

## 为什么需要RAG？

### 问题：AI的"幻觉"

大语言模型有一个严重问题——**幻觉（Hallucination）**：
- 模型会自信地编造不存在的事实
- 知识截止日期后的事情不知道
- 无法访问私有数据

**例子**：
```
用户：2025年诺贝尔物理学奖得主是谁？
AI（无RAG）：[可能编造一个不存在的名字]
AI（有RAG）：[检索后] 根据最新信息，2025年诺贝尔物理学奖...
```

### RAG如何解决？

```
用户问题 → 检索相关文档 → 将文档+问题一起给AI → AI基于文档回答
```

## RAG的工作原理

### 基本流程

```
┌─────────────┐
│  用户提问   │
└──────┬──────┘
       ↓
┌──────────────┐
│  文本嵌入    │  将问题转为向量
│  (Embedding) │
└──────┬───────┘
       ↓
┌──────────────┐
│  向量检索    │  在知识库中搜索相似内容
│  (Retrieval) │
└──────┬───────┘
       ↓
┌──────────────┐
│  上下文拼接  │  将检索结果+问题组合
└──────┬───────┘
       ↓
┌──────────────┐
│  LLM生成回答 │  基于检索到的信息回答
│  (Generation)│
└──────────────┘
```

### 关键技术组件

#### 1. Embedding（文本嵌入）
- 将文本转换为向量表示
- 语义相似的文本，向量也相似
- 常用模型：OpenAI text-embedding-3, BGE, E5

#### 2. Vector Database（向量数据库）
- 存储和检索向量
- 支持相似度搜索
- 常用：Pinecone, Weaviate, Milvus, Chroma, FAISS

#### 3. Chunking（文档分块）
- 将长文档切分为小块
- 每个块独立嵌入和检索
- 需要平衡块大小和上下文完整性

#### 4. Retrieval（检索）
- 根据问题向量搜索最相关的文档块
- 通常返回Top-K个结果

#### 5. Generation（生成）
- 将检索到的文档和问题一起给LLM
- LLM基于这些信息生成回答

## RAG的演进

### 1. Naive RAG（朴素RAG）
最基本的实现：
- 文档分块 → 嵌入 → 检索 → 生成
- **问题**：检索质量不稳定，可能检索到不相关内容

### 2. Advanced RAG（高级RAG）
改进检索质量：
- **查询改写**：优化用户问题
- **混合检索**：向量搜索 + 关键词搜索
- **重排序**：对检索结果重新排序
- **上下文压缩**：去除不相关信息

### 3. Modular RAG（模块化RAG）
灵活组合不同组件：
- 多种检索策略
- 多种嵌入模型
- 自适应检索

### 4. Graph RAG
使用知识图谱：
- 将文档构建为知识图谱
- 基于图结构检索
- 适合复杂关系查询
- **代表**：Microsoft GraphRAG

## RAG vs Fine-tuning

| 维度 | RAG | Fine-tuning |
|------|-----|-------------|
| 原理 | 检索外部知识 | 修改模型参数 |
| 实时性 | ✅ 可访问最新数据 | ❌ 知识固定 |
| 成本 | 低（无需训练） | 高（需要GPU训练） |
| 幻觉 | 减少 | 可能仍有 |
| 私有数据 | ✅ 支持 | ⚠️ 需要注意隐私 |
| 复杂推理 | ⚠️ 较弱 | ✅ 更强 |

**最佳实践**：两者结合使用

## RAG的实际应用

### 1. 企业知识库
- 内部文档检索
- 客服知识库
- 政策法规查询

### 2. 客户服务
- 基于产品文档回答
- 准确性大幅提升

### 3. 法律研究
- 检索法律条文和判例
- 辅助法律分析

### 4. 学术研究
- 文献检索和总结
- 论文写作辅助

### 5. 代码助手
- 基于代码库回答问题
- API文档查询

## RAG的最佳实践

### 1. 文档预处理
- 清洗数据，去除噪声
- 合理分块，保持语义完整
- 添加元数据（来源、时间、类别）

### 2. 检索优化
- 使用混合检索（向量+关键词）
- 重排序提高相关性
- 适当增加检索数量

### 3. 生成优化
- 明确引用来源
- 指示模型基于检索内容回答
- 处理"无相关信息"的情况

### 4. 评估
- 检索准确率
- 回答质量
- 幻觉率

---

## RAG的挑战

1. **检索质量**：检索到不相关内容会导致错误回答
2. **上下文长度限制**：检索内容过多会超出上下文窗口
3. **实时性**：需要定期更新索引
4. **复杂推理**：对于需要多步推理的问题效果有限
5. **成本**：向量数据库和嵌入API的成本

---

## 未来展望

1. **长上下文替代**？：随着上下文窗口增长（200万token），简单场景可能直接塞入全文
2. **Graph RAG成熟**：知识图谱+RAG的结合
3. **多模态RAG**：图像、视频的检索
4. **自适应RAG**：根据问题类型自动选择策略
5. **端到端优化**：检索和生成联合训练
