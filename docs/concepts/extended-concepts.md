# 进阶概念 {#extended-concepts}

除了核心概念外，AI大模型生态中还有许多重要的进阶概念，理解它们能帮助你更深入地掌握AI技术栈。

---

## Agent相关概念

### Hook（钩子）

**定义**：Agent执行过程中的生命周期回调函数，在特定时机触发。

**类比**：就像Web开发中的中间件，在请求处理前后插入自定义逻辑。

**常见Hook时机**：
```
Agent执行流程：
  on_start          → Agent开始执行
  before_tool_call  → 调用工具前
  after_tool_call   → 调用工具后
  before_response   → 生成回复前
  after_response    → 生成回复后
  on_error          → 出错时
  on_end            → Agent结束
```

**用途**：
- 日志记录：记录Agent的每一步操作
- 安全检查：在工具调用前验证权限
- 成本监控：追踪token消耗
- 结果验证：检查输出是否符合预期

**示例**（LangChain）：
```python
from langchain.callbacks import StdOutCallbackHandler

handler = StdOutCallbackHandler()
agent.run("查询销售数据", callbacks=[handler])
```

---

### Skill（技能）

**定义**：Agent的可复用能力单元，封装了完成特定任务所需的指令、工具和知识。

**类比**：就像手机App——每个App解决一个问题，Skill让Agent具备特定能力。

**Skill的组成**：
```
Skill
├── 指令（Instructions）：如何执行任务
├── 工具（Tools）：需要调用的外部功能
├── 知识（Knowledge）：领域特定信息
└── 示例（Examples）：参考案例
```

**常见Skill类型**：
| Skill | 功能 | 工具 |
|-------|------|------|
| 代码审查 | 审查PR质量 | GitHub, Git |
| 数据分析 | 分析数据集 | Python, SQL |
| 文档生成 | 生成技术文档 | 文件系统 |
| 客户服务 | 处理客户问题 | CRM, 知识库 |
| 研究助手 | 搜索和总结 | 搜索, 爬虫 |

**Skill vs Function Calling**：
| 维度 | Skill | Function Calling |
|------|-------|------------------|
| 粒度 | 高层能力 | 底层函数 |
| 组合 | 包含多个工具 | 单个工具 |
| 知识 | 包含领域知识 | 无 |
| 示例 | 包含参考案例 | 无 |

---

### Tool（工具）

**定义**：Agent可以调用的外部功能，是Agent与外界交互的基本单元。

**与Function Calling的关系**：
- Function Calling是机制（如何调用）
- Tool是被调用的对象（调用什么）

**Tool的设计原则**：
1. **单一职责**：一个Tool只做一件事
2. **清晰描述**：让AI理解何时使用
3. **错误处理**：返回有意义的错误信息
4. **幂等性**：重复调用结果一致

---

### Memory（记忆）

**定义**：Agent保持上下文和学习的能力。

**记忆类型**：
| 类型 | 说明 | 持久性 | 例子 |
|------|------|--------|------|
| 短期记忆 | 当前对话上下文 | 会话内 | 对话历史 |
| 长期记忆 | 跨会话知识 | 永久 | 用户偏好 |
| 工作记忆 | 当前任务状态 | 任务内 | 中间结果 |
| 情景记忆 | 过去的经历 | 永久 | 历史交互 |
| 语义记忆 | 一般知识 | 永久 | 事实知识 |

**实现方式**：
- **向量数据库**：存储和检索记忆
- **摘要**：压缩长对话历史
- **实体提取**：记住关键信息

---

### Workflow（工作流）

**定义**：多个步骤组成的Agent执行流程，实现复杂的多步骤任务。

**与简单Agent的区别**：
| 维度 | 简单Agent | Workflow |
|------|-----------|----------|
| 步骤 | 单步或简单循环 | 多步骤，有条件分支 |
| 控制 | Agent自主 | 预定义流程 |
| 可靠性 | 较低 | 较高 |
| 适用 | 简单任务 | 复杂业务流程 |

**Workflow模式**：
```
顺序执行：A → B → C
条件分支：if X then A else B
并行执行：A || B || C
循环：while condition do A
人工审批：A → [人工审批] → B
```

**代表工具**：
- LangGraph：基于图的Workflow
- Prefect：数据工作流
- Temporal：可靠工作流

## RAG相关概念

### Embedding（嵌入/向量化）

**定义**：将文本、图像等转换为数值向量的技术。

**核心思想**：语义相似的内容，向量也相似。

**示例**：
```
"猫" → [0.2, 0.8, 0.1, ...]
"小猫" → [0.21, 0.79, 0.11, ...]  （相似）
"汽车" → [0.9, 0.1, 0.7, ...]    （不相似）
```

**常用模型**：
| 模型 | 维度 | 特点 |
|------|------|------|
| OpenAI text-embedding-3 | 1536/3072 | 商用最强 |
| BGE (BAAI) | 1024 | 开源最强 |
| E5 (Microsoft) | 1024 | 开源 |
| GTE (阿里) | 1024 | 中文优化 |

---

### Vector Database（向量数据库）

**定义**：专门存储和检索向量的数据库。

**核心功能**：根据向量相似度快速找到最相关的内容。

**为什么需要**：
- 传统数据库无法高效进行相似度搜索
- 向量数据库针对高维向量优化

**主流产品**：
| 数据库 | 类型 | 特点 |
|--------|------|------|
| Pinecone | 托管 | 最易用 |
| Weaviate | 开源 | 功能丰富 |
| Milvus | 开源 | 高性能 |
| Chroma | 开源 | 轻量级 |
| Qdrant | 开源 | Rust实现 |
| FAISS | 库 | Facebook出品 |

---

### Chunking（分块）

**定义**：将长文档切分为小块，便于嵌入和检索。

**为什么重要**：
- Embedding模型有token限制
- 太大的块会稀释语义
- 太小的块缺少上下文

**分块策略**：
| 策略 | 说明 | 适用 |
|------|------|------|
| 固定大小 | 按字符/token数 | 通用 |
| 语义分块 | 按段落/章节 | 文档 |
| 递归分块 | 逐级细分 | 复杂文档 |
| 基于结构 | 按HTML/Markdown结构 | 网页 |

**最佳实践**：
- 块大小：200-1000 token
- 重叠：10-20%
- 保留元数据（来源、位置）

---

### Reranking（重排序）

**定义**：对初步检索结果重新排序，提高相关性。

**为什么需要**：
- 向量搜索可能返回不太相关的结果
- 重排序用更精确的模型筛选

**流程**：
```
用户问题 → 向量检索（Top 20）→ 重排序 → Top 5 → 生成回答
```

**常用模型**：
- Cohere Rerank
- BGE Reranker
- Cross-encoder

---

### Query Transformation（查询转换）

**定义**：优化用户查询以提高检索质量。

**技术**：
| 技术 | 说明 |
|------|------|
| 查询改写 | 用不同方式表达同一问题 |
| 查询扩展 | 添加相关关键词 |
| HyDE | 先生成假设性答案，再用答案检索 |
| 多查询 | 将一个问题拆分为多个子查询 |

## 模型相关概念

### Token（Token）

**定义**：文本处理的基本单位。

**Token化规则**：
- 英文：约1个token ≈ 4个字符 ≈ 0.75个单词
- 中文：约1个token ≈ 1-2个汉字
- 代码：变量名、符号各算一个token

**示例**：
```
"Hello world" → ["Hello", " world"] → 2 tokens
"人工智能" → ["人工", "智能"] → 2 tokens
```

**为什么重要**：
- 模型按token计费
- 上下文窗口以token为单位
- 影响响应速度

---

### Context Window（上下文窗口）

**定义**：模型一次能处理的最大token数量。

**各模型上下文**：
| 模型 | 上下文 |
|------|--------|
| GPT-5 | 256K |
| Claude 4 | 200K |
| Gemini 2.5 Pro | 2M |
| LLaMA 4 Scout | 10M |
| DeepSeek R1 | 128K |

**挑战**：
- 超长上下文增加成本
- 注意力分散（"大海捞针"问题）
- 需要平衡长度和相关性

---

### Temperature（温度）

**定义**：控制模型输出随机性的参数。

**取值范围**：0-2

| 值 | 效果 | 适用场景 |
|----|------|----------|
| 0 | 确定性输出 | 代码生成、数据提取 |
| 0.3-0.7 | 平衡 | 通用对话 |
| 0.8-1.2 | 创意 | 创意写作 |
| 1.5+ | 高随机 | 头脑风暴 |

---

### Top-P（核采样）

**定义**：另一种控制输出多样性的方式。

**原理**：只从累积概率达到P的token中采样。

**与Temperature的关系**：
- 两者都控制随机性
- 通常只调一个
- Top-P更直观

---

### System Prompt（系统提示）

**定义**：定义模型行为和角色的提示词。

**示例**：
```
你是一位专业的Python开发者。
- 始终使用类型注解
- 遵循PEP 8规范
- 编写清晰的docstring
- 优先使用标准库
```

**最佳实践**：
- 明确角色定位
- 设定行为边界
- 指定输出格式
- 提供参考示例

---

### Streaming（流式输出）

**定义**：逐token返回结果，而非等待完整响应。

**优势**：
- 用户体验更好（即时反馈）
- 感觉响应更快
- 可以提前中断

**实现**：
```python
stream = client.chat.completions.create(
    model="gpt-4o",
    messages=[...],
    stream=True
)
for chunk in stream:
    print(chunk.choices[0].delta.content, end="")
```

---

### Structured Output（结构化输出）

**定义**：让模型输出特定格式（如JSON）。

**方式**：
| 方式 | 说明 |
|------|------|
| JSON Mode | 强制输出JSON |
| Function Calling | 输出函数调用参数 |
| JSON Schema | 按Schema输出 |
| XML标签 | 用标签结构化 |

**示例**：
```json
{
  "name": "张三",
  "age": 25,
  "skills": ["Python", "AI"]
}
```

## 安全与评估概念

### Guardrails（护栏）

**定义**：限制AI行为的安全机制。

**类型**：
| 类型 | 说明 | 示例 |
|------|------|------|
| 输入过滤 | 阻止有害输入 | 检测敏感词 |
| 输出验证 | 检查输出合规性 | 格式验证 |
| 行为限制 | 限制Agent权限 | 只读操作 |
| 内容审核 | 检测不当内容 | 有害信息检测 |

**工具**：
- Guardrails AI
- NeMo Guardrails (NVIDIA)
- LangChain Guardrails

---

### Grounding（接地/事实锚定）

**定义**：将AI输出与真实数据源关联。

**为什么重要**：
- 减少幻觉
- 提高可信度
- 支持事实核查

**实现方式**：
- RAG：基于检索到的事实回答
- 引用来源：标注信息出处
- 事实验证：交叉验证信息

---

### Human-in-the-Loop（人机协作）

**定义**：在AI流程中加入人工审核和干预。

**应用场景**：
| 场景 | 说明 |
|------|------|
| 高风险决策 | 人工确认后执行 |
| 质量审核 | 人工检查AI输出 |
| 反馈学习 | 人工标注改进模型 |
| 异常处理 | AI无法处理时转人工 |

**实现方式**：
- 审批流程：AI提议→人工批准→执行
- 人工反馈：对AI输出评分
- 交互式修正：人工修改AI结果

---

### Evaluation（评估）

**定义**：衡量AI模型性能的方法和标准。

**主流基准**：
| 基准 | 测试内容 | 说明 |
|------|----------|------|
| MMLU | 多任务理解 | 57个学科 |
| HumanEval | 代码生成 | Python编程 |
| SWE-bench | 真实编程 | GitHub issue修复 |
| ARC-AGI | 抽象推理 | 通用智能 |
| GPQA | 研究生科学 | 科学推理 |
| MATH | 数学推理 | 数学竞赛题 |
| AIME | 数学竞赛 | 高难度数学 |
| LMArena | 用户偏好 | 人类投票 |

## 新兴概念（2025-2026）

### Context Engineering（上下文工程）

**定义**：设计和优化AI上下文的系统方法。

**比Prompt Engineering更广**：
- Prompt Engineering：设计单个提示
- Context Engineering：设计整个上下文系统

**包含**：
- 系统提示设计
- 工具定义
- 知识库管理
- 记忆策略
- 示例选择

---

### Vibe Coding（氛围编程）

**定义**：用自然语言描述需求，让AI生成代码的编程方式。

**特点**：
- 不需要深入理解代码细节
- 快速原型开发
- 依赖AI的代码质量

**工具**：Cursor, Claude Code, Windsurf

**争议**：
- 支持者：降低编程门槛
- 批评者：代码质量难以保证

---

### Agentic AI（智能体AI）

**定义**：能够自主感知、决策、行动的AI系统范式。

**与传统AI的区别**：
| 维度 | 传统AI | Agentic AI |
|------|--------|------------|
| 交互 | 被动响应 | 主动行动 |
| 能力 | 单一任务 | 多步骤任务 |
| 工具 | 不使用 | 广泛使用 |
| 自主性 | 低 | 高 |

---

### Multi-Agent（多智能体）

**定义**：多个Agent协作完成任务。

**模式**：
| 模式 | 说明 | 示例 |
|------|------|------|
| 主从模式 | 一个主Agent协调 | 项目经理+开发者 |
| 对等模式 | Agent平等协作 | 代码审查 |
| 流水线 | 顺序处理 | 数据采集→分析→报告 |
| 辩论模式 | Agent讨论达成共识 | 方案评审 |

**框架**：CrewAI, AutoGen, LangGraph

---

### A2A（Agent-to-Agent Protocol）

**定义**：Google提出的Agent间通信协议（2025年4月）。

**与MCP的关系**：
- MCP：Agent ↔ 工具/数据
- A2A：Agent ↔ Agent

**目的**：让不同厂商的Agent可以互相协作。

---

### Speculative Decoding（推测解码）

**定义**：用小模型快速生成草稿，大模型验证的推理加速技术。

**原理**：
```
小模型快速生成5个token → 大模型并行验证 → 接受正确的token
```

**效果**：推理速度提升2-3倍，输出质量不变。

---

### Synthetic Data（合成数据）

**定义**：用AI生成的训练数据。

**用途**：
- 数据增强
- 替代稀缺数据
- 微调模型

**风险**：
- 模型坍塌（model collapse）
- 偏见放大
- 质量控制

---

### Model Router（模型路由器）

**定义**：根据任务自动选择最合适的模型。

**示例**：
```
简单问题 → GPT-4o mini（便宜）
复杂推理 → o3-pro（强大）
编程任务 → Claude 4 Opus（擅长编程）
长文档 → Gemini 2.5 Pro（上下文长）
```

**优势**：平衡成本和效果。

---

### AI Gateway（AI网关）

**定义**：集中管理AI访问的中间层。

**功能**：
- 统一API接口
- 负载均衡
- 成本控制
- 安全审计
- 缓存

**代表**：Azure API Management, LiteLLM
